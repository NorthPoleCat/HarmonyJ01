# Japanese01 工程分析报告

## 1. 项目概览

该工程是一个基于 **HarmonyOS Stage Model + ArkTS + ArkUI** 开发的移动端学习应用，应用名称为 **“新版标准日本语-初级”**，包名为 `com.polarcat.japanese01`，当前版本号为 `1.2.0`。

从目录结构和实现方式来看，这是一个以 **本地静态教学资源驱动** 的单机应用，核心目标是围绕《新版标准日本语-初级》提供：

- 课文目录浏览
- 本地 HTML 课文展示
- 词汇详情查看
- 句型/例句详情查看
- 语法跳转
- 音频片段播放
- 用户偏好设置
- 首次隐私确认

项目明显偏向“内容型 App”架构，代码本身相对轻量，主要复杂度集中在大量本地资源的组织、WebView 与 ArkTS 的桥接，以及单词变形规则的内置实现。

## 2. 技术栈与工程形态

### 2.1 技术栈

- UI 框架：ArkUI（声明式）
- 语言：ArkTS / ETS
- 应用模型：Stage Model
- Web 容器：`@kit.ArkWeb`
- 数据存储：`@kit.ArkData` Preferences
- 音频播放：`@kit.MediaKit` AVPlayer
- 构建工具：Hvigor
- 测试框架：Hypium

### 2.2 工程形态

这是一个 **单应用、单主模块** 的 HarmonyOS 工程：

- 根工程负责应用级配置与签名、产品定义
- `entry` 为唯一业务模块
- 页面、组件、模型、工具均集中在 `entry/src/main/ets`
- 教学内容主要位于 `entry/src/main/resources/rawfile`

工程内没有后端调用、没有网络请求、也没有数据库层，属于典型的 **离线内容分发型应用**。

## 4. 页面与业务流转

当前工程共注册了 **6 个页面**：

- `pages/PrivacyDialogPage`
- `pages/Index`
- `pages/WebViewPage`
- `pages/WordView`
- `pages/SentenceView`
- `pages/SettingsView`

### 4.1 启动流程

应用入口为 `EntryAbility`。在 `onWindowStageCreate` 中，默认加载的不是首页，而是：

- 首次进入时先打开 `PrivacyDialogPage`
- 如果用户已经同意隐私协议，则自动跳转到 `Index`

这说明应用启动链路是：

`EntryAbility -> PrivacyDialogPage -> Index`

### 4.2 首页 Index

`Index.ets` 是课程目录页，启动时会从 `rawfile/allLessons.json` 读取课程结构并解析为：

- `LessonSection`
- `LessonClass`

页面以分组列表形式展示单元与课文，点击后跳转到 `WebViewPage`，并传递：

- HTML 文件路径，如 `Lessons/Lesson01.html`
- 页面标题

这一页本质上是整个应用的 **内容分发入口**。

### 4.3 WebViewPage

`WebViewPage.ets` 是工程中最核心的业务枢纽，职责包括：

- 加载本地 HTML 教材页面
- 注册 JS Bridge
- 处理语法、词汇、句子、音频等交互
- 提供返回、首页、设置入口
- 在深色模式下注入额外 CSS

这里采用的是 **“本地 HTML + JS 事件 + ArkTS 处理业务”** 的混合架构。也就是说，教学内容主要写在 HTML 中，而原生层负责承接交互能力。

### 4.4 WordView

`WordView.ets` 用于展示词汇详情，包括：

- 单词本体
- 罗马音
- 中文释义
- 词性标签
- 自动词/他动词标签
- 副词属性
- 各种活用变形

该页的主要复杂度在 `deformation()` 方法中。当前实现把大量日语变形规则直接写成代码分支，覆盖：

- 五段动词
- 一段动词
- カ变、サ变
- 一类形容词、二类形容词
- 特殊词条，如 `ある`、`行く`、`いい`

这是一种直接、可控但维护成本较高的实现方式。

### 4.5 SentenceView

`SentenceView.ets` 用于展示句子详情，包含：

- 原句
- 翻译
- 词组提示
- 涉及语法点

其中语法点可继续跳转回 `WebViewPage` 打开对应语法 HTML。

### 4.6 SettingsView

`SettingsView.ets` 是设置页，主要功能是通过 Toggle 控制 Web 内容样式开关：

- 标注名词
- 标注动词
- 标注形容词
- 标注中文
- 标注假名

设置项通过 Preferences 持久化，变更后调用 `controller.refresh()` 刷新预览页。页面中还内嵌了一个 `example.html` 作为样式预览载体。

## 5. 核心架构设计

### 5.1 总体架构

该工程可以理解为 4 层：

1. 展示层：ArkUI 页面与组件
2. 内容层：本地 HTML / JSON / 音频 / CSS / JS 资源
3. 桥接层：`JSBridgeClass`
4. 能力层：Preferences、AVPlayer、Router

### 5.2 JS Bridge 是架构核心

`JSBridgeClass.ets` 是全工程最关键的中枢之一，承担了 Web 内容和原生能力之间的桥接：

- `grammar(param)`：切换语法 HTML
- `sentence(param)`：读取句子 JSON 并跳转到句子详情页
- `word(param)`：读取词汇 JSON 并跳转到词汇详情页
- `sentencePlay(param)`：播放例句片段音频
- `audioPlay(param)`：播放课文片段音频
- `audioReset()`：停止音频
- `audioRecycle(param)`：设置循环播放
- `loadCss()`：根据偏好注入不同样式

这意味着：

- HTML 页面负责定义内容和触发事件
- ArkTS 负责读取资源、控制跳转、调用系统能力

这种设计很适合内容量大、页面格式高度固定的学习类应用。

### 5.3 音频能力

`AudioPlayer.ets` 采用单例模式封装 `media.AVPlayer`，支持：

- 加载 rawfile 音频
- 指定起止时间播放片段
- 文章音频暂停/继续
- 句子音频片段播放
- 循环播放
- 播放完成后回调重置 UI

从逻辑上看，应用并不是播放整段音频，而是依赖：

- `audioArticle.json`
- `senAudio.json`

这两个映射文件来确定音频文件名及起止时间。

### 5.4 数据模型

模型层相对简单，主要作为 JSON 映射和 UI 展示辅助：

- `LessonSection` / `LessonClass`
- `WordModel`
- `SentenceModel`
- `WordTypeClass`
- `DeformationClass`
- `JSBridgeClass`

其中 `WordTypeClass` 与 `DeformationClass` 更接近“领域枚举封装器”，承担：

- 标签文案
- 颜色映射
- 语法跳转地址映射

## 6. 资源组织分析

该工程的数据资源规模较大，是项目最有分量的部分。

### 6.1 资源规模

`entry/src/main/resources/rawfile` 目录约 **253MB**，核心内容包括：

- 课文 HTML：62 个
- 语法 HTML：147 个
- 句子 JSON：901 个
- 词汇 JSON：10 个
- 音频文件：48 个
- 音频映射 JSON：2 个
- 公共样式 / 脚本文件若干

此外，`SentenceJsons` 下共有 **49 个子目录**，说明句子资源是按课次或主题拆分管理的。

### 6.2 内容组织方式

资源命名具有较强规律性，例如：

- `Lessons/Lesson01.html`
- `grammar/grammar14-01.html`
- `SentenceJsons/14/14-21.json`
- `wordJsons/noun/noun-0.json`
- `audio/14.mp3`

这种方式的优点是：

- 便于离线打包
- 便于按规则推导文件路径
- 不依赖数据库或服务端索引

缺点是：

- 路径规则和编号规则耦合在代码中
- 一旦资源重命名，代码容易失效
- 缺少统一索引层时，数据校验成本较高

## 7. 组件与复用性

当前只有 **2 个通用组件**：

- `WordPartCell`
- `DeformationCell`

说明项目整体仍以页面直接组织 UI 为主，组件化程度不高，但对于当前体量来说仍然可维护。  
如果未来准备扩展更多课程、词法卡片、练习页、收藏页，建议进一步抽离：

- 通用顶部导航栏
- 通用标签组件
- 通用详情分组组件
- 通用 Web 容器页

## 8. 构建与配置分析

### 8.1 应用配置

应用级配置表明：

- Bundle Name：`com.polarcat.japanese01`
- Vendor：`polarcat`
- Version Name：`1.2.0`
- 目标设备：`phone`
- SDK：`6.0.0(20)`

### 8.2 构建体系

项目使用 Hvigor 默认插件：

- 根工程：`appTasks`
- `entry` 模块：`hapTasks`

没有自定义构建插件，也没有复杂流水线逻辑，构建系统比较标准。

### 8.3 混淆与资源设置

`entry/build-profile.json5` 中：

- `copyCodeResource.enable = false`
- release 模式定义了混淆规则文件
- 但 `obfuscation.ruleOptions.enable = false`

也就是说，当前 release 包虽然配置了混淆文件路径，但实际上 **没有启用混淆**。

## 9. 测试现状

工程包含两类测试：

- `src/test` 本地单元测试
- `src/ohosTest` 设备侧测试

但当前内容基本都是模板示例，只验证字符串包含/相等等基础断言，没有覆盖真实业务逻辑，例如：

- 路由跳转
- 词汇数据解析
- 句子数据解析
- 音频播放边界
- 偏好设置读写
- JS Bridge 入参与异常处理

结论是：**项目具备测试目录，但尚未形成有效测试体系**。

## 10. 当前实现的优点

### 10.1 架构思路清晰

采用“本地 HTML 内容 + ArkTS 原生壳 + JS Bridge”的模式，非常适合教材型应用，既能快速沉淀内容，也能利用原生能力完成路由、音频和设置管理。

### 10.2 离线能力强

课程、语法、句子、词汇、音频均为本地资源，理论上不依赖网络，用户体验稳定，适合学习工具场景。

### 10.3 数据组织规律强

资源文件的路径和编号命名比较统一，桥接层可以通过规则直接定位文件，减少了额外索引层的复杂度。

### 10.4 页面链路简洁

页面数量不多，用户核心路径明确：

隐私确认 -> 课程目录 -> 课文浏览 -> 词汇/句子/语法/音频

## 11. 主要问题与风险点

### 11.1 签名敏感信息直接写入仓库

`build-profile.json5` 中直接包含：

- `storeFile`
- `storePassword`
- `keyAlias`
- `keyPassword`
- `profile`
- `certpath`

这是当前工程中最需要优先处理的问题。即使密码经过某种形式编码，只要随仓库分发，依然属于高风险做法。  
建议尽快改为：

- 本地私有配置文件
- CI 环境变量注入
- 不入库的签名材料路径配置

### 11.2 词形变化逻辑硬编码严重

`WordView.ets` 中包含大量分支式活用推导逻辑，优点是直接，缺点是：

- 文件很长
- 可测试性一般
- 扩展新规则容易引入回归
- 某些特殊词条修正需要继续堆 `if`

后续更推荐把变形规则抽到独立服务或规则表中。

### 11.3 Web 与原生耦合较深

HTML 文件、CSS、JS、ArkTS、资源命名之间存在明显的路径和命名耦合。  
一旦资源侧做批量重命名，桥接逻辑、HTML 事件名、CSS 注入路径都可能同步失效。

### 11.4 异常处理偏弱

当前虽然对部分资源读取失败做了日志打印，但整体仍偏“失败即控制台输出”，例如：

- JSON 解析失败
- 路径推导失败
- WebView 页面加载失败
- 音频文件不存在

用户侧缺少明确反馈，也缺少降级策略。

### 11.5 设置项扩展性一般

设置页当前直接写了多组重复 Toggle 代码，随着配置项增多，可维护性会下降。  
后续可改成配置数组驱动渲染。

### 11.6 测试缺失导致回归风险高

项目核心价值主要在内容跳转和规则正确性，但这些目前都没有自动化测试保护。

## 12. 建议的优化方向

### 12.1 优先级 P0

- 清理仓库中的签名敏感配置
- 检查是否已泄露签名文件与密码材料
- 将签名改成本地环境或 CI 注入

### 12.2 优先级 P1

- 把 `WordView` 的变形逻辑拆到独立文件
- 为词形变换建立单元测试
- 为 `JSBridgeClass` 建立资源路径和参数解析测试
- 对资源缺失场景增加统一错误提示

### 12.3 优先级 P2

- 抽离通用顶栏组件
- 设置页改为配置驱动
- 引入统一资源索引表，减少路径硬编码
- 增加资源完整性校验脚本

### 12.4 优先级 P3

- 为课文、语法、句子建立元数据索引
- 支持收藏、最近学习、搜索等功能
- 对 HTML 内容体系进行模块化整理

## 13. 适合后续接手者的理解方式

如果后续要继续维护该项目，建议按下面顺序理解：

1. 先看 `EntryAbility` 和 `PrivacyDialogPage`，理解应用入口
2. 再看 `Index`，理解课程目录如何由 `allLessons.json` 驱动
3. 重点看 `WebViewPage + JSBridgeClass`，这是主业务核心
4. 再看 `WordView` 和 `SentenceView`，理解详情页数据流
5. 最后看 `rawfile` 目录结构，建立资源编号规则认知

## 14. 总结

这是一个结构相对清晰、产品目标明确的 HarmonyOS 日语学习应用。  
它最大的特点不是复杂业务系统，而是：

- 以本地资源为中心
- 以 WebView 为内容承载
- 以 ArkTS 原生能力补齐交互

从工程角度看，这个项目已经具备稳定可用的内容型应用雏形，核心优势是离线化和内容沉淀；核心短板则是签名信息安全、测试不足以及部分逻辑硬编码。  
如果后续继续迭代，建议优先处理安全问题与规则模块化问题，这两项会最直接影响项目的长期维护质量。
